Filter Settings UI & Functionality
The interactive part of the filter nodes allows users to adjust settings and see real-time previews. Here's how it works:

1. Filter Settings Definition
First, each filter type has its settings defined in filters.ts:

export function getFilterSettings(filterType: string): FilterSetting[] {
  switch (filterType) {

      return [

    // Other filter definitions...
  }
}
2. Settings UI Components
Inside each FilterNode, the settings are rendered based on their type:

{filterSettings.map(setting => (

3. Setting Change Handler
When a user adjusts a setting via slider or select, this handler updates the state and triggers a preview update:

// From FilterNode.tsx
const handleSettingChange = (name: string, value: any) => {
  const newSettings = { ...settings, [name]: value };
  setSettings(newSettings);
  
  // This is important - it communicates the setting change to the parent
  if (data.onSettingsChange) {
    data.onSettingsChange(id, newSettings);
  }
};
4. The Update Chain
The full sequence when a user moves a slider:

User drags a slider (or selects a dropdown option)
handleSettingChange is called with the new value
Local node state is updated with the new settings value
The onSettingsChange callback from the node's data prop is triggered
This callback is defined in useFilterGraph.ts when the node is created:
// When creating a filter node:
const newNode: Node = {
  id,
  type: 'filterNode',
  position,
  data: {
    type: filterType as FilterType,
    label: `${capitalizedType}`,
    settings: {},
    onSettingsChange: (nodeId: string, settings: Record<string, any>) => {
      // Update the node data with new settings
      updateNodeData(nodeId, { settings });
      
      // Clear cached results for this node and any downstream nodes
      clearDownstreamCache(nodeId);
    }
  }
};
The updateNodeData function updates the node's settings in the global state
clearDownstreamCache removes cached image data for this node and all nodes that depend on it
The next time the UI renders, it will recalculate the preview with the new settings
5. The Preview Update
Since the cache was cleared, the next render cycle triggers:

The node's preview display calls getProcessedImage to get its current output
Since the cache was cleared, it reprocesses the input image with the new settings
The filter processing uses the new settings value when applying the effect:
// In the filter implementation
function applyBlurFilter(
  ctx: CanvasRenderingContext2D,
  width: number,
  height: number,
  settings: Record<string, any>
) {
  // Use the radius from settings, or fallback to default
  const radius = settings.radius || 5;
  
  // Apply the filter with the current setting value
  ctx.filter = `blur(${radius}px)`;
  // ... rest of implementation
}
The new processed image is cached and set as the node's preview
The UI updates to show the new preview image
This gives users a seamless, real-time experience where they can adjust settings and immediately see the results in both the node preview and any downstream node previews or the final output.
