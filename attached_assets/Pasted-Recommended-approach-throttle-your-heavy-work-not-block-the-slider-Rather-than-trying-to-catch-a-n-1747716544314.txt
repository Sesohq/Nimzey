Recommended approach: throttle your heavy work, not block the slider
Rather than trying to catch a non-existent “commit” event, wrap your preview trigger in a throttle so you only re-render at most, say, once every 100 ms:

ts
Copy
Edit
// at the top of FilterNode.tsx
import { throttle } from 'lodash';
const throttledRefresh = useMemo(
  () => throttle(onPaneClick!, 100, { leading: true, trailing: true }),
  [onPaneClick]
);

// ...

<CustomSlider
  value={[param.value]}
  min={param.min}
  max={param.max}
  step={param.step}
  onValueChange={([v]) => {
    // update node data immediately so the thumb moves
    handleParamChange(param.id, v);
    // but only redraw at most every 100 ms
    throttledRefresh();
  }}
  disabled={!data.enabled || param.isConnected}
/>
Why this works better than commit

The slider UI remains rock-solid because you never block the internal pointer events.

You still get real-time feedback, but capped at 10 renders / sec instead of 60+.

No reliance on onValueCommit.

// draw at a capped size
export async function loadImageData(src: string, maxSize: number) {
  const img = new Image();
  img.src = src;
  await img.decode();

  // compute scaled dimensions
  const scale = Math.min(1, maxSize / Math.max(img.width, img.height));
  const w = Math.round(img.width * scale);
  const h = Math.round(img.height * scale);

  const tmp = document.createElement('canvas');
  tmp.width = w; tmp.height = h;
  const ctx = tmp.getContext('2d')!;
  ctx.drawImage(img, 0, 0, w, h);
  return ctx.getImageData(0, 0, w, h);
}

Putting it all together
Throttle your onPaneClick in the slider’s onValueChange so the slider never locks up.

Cap your preview pipeline’s resolution to 360px (or even 144px) by passing maxSize.

This combo will give you:

Silky-smooth slider control,

Immediate visual feedback,

And previews that only do a tiny fraction of the work of a full-res render.
