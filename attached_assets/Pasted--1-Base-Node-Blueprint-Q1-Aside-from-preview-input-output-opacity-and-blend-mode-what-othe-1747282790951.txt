🧱 1. Base Node Blueprint
Q1: Aside from preview, input, output, opacity, and blend mode, what other global properties should every filter node have by default? Each filter node should be togglable on/off. Properties that an intermediate node has that a destination node does NOT have, will be ignored when the intermediate node is disabled and default values will be applied to fields they do not share. Color tagging should be optional, default to grey but having a drop down in the header to specify which color is selected. Labeling may hurt our coding since we describe in coding inputs, so no labelling. UI state can have collapsed options, no need for extension beyond what we’d set as the nodes defaults. 
(e.g., enabled toggle, color tagging, label, collapsed/expanded UI state?)
Q2: Should every filter node be required to define:
A default preview image size or resolution? The preview image within the node is always within the container, and small/low resolution, just to share a preview. 


A default value for each input setting? Yes, that way each node has a preview attached to it on initial load. 


A blend mode or just default to “normal”? Normal should be default. 



🔄 2. Adjustment Input Containers (Linkable Parameters)
Q3: Are all parameters within a node (like “Strength,” “Size,” “Texture”) assumed to be linkable, or do some need to be locked/static? Yes, with define further for specific nodes.
Q4: Should each of these parameter containers:
Have a unique ID so it can be linked directly? yes


Have its own input/output typing? yes


Store its control UI style? (slider, dropdown, image picker, etc.) yes


Q5: Will these parameters also need a preview screen for their input links?
 (E.g., previewing the texture input coming into the “Size” slider?) no need to double it up.

🔁 3. Input/Output Typing System
Q6: What are the main types you foresee? We will build this in tandem but let's start with 3 basic inputs and outputs.
 Examples:
image, mask, float, color, vector2, texture, boolean, blendSettings
 (You can invent types to help enforce logic)


Q7: Can a node output multiple types at once?
 (e.g., a Blur node outputs both an image and a mask) - Yes it can, but also it may accept multiple types as well. 
Q8: Do you want to support fallbacks or type casting? We do not convert it, it’s what we’d want to avoid and result in user confusion. What we do is that when a user tries to apply the wrong type into a node, we display an error message that it cannot accept “that” specific data property, and suggest what properties are possible. Let’s actually make it so if there is a new node dropped on the canvas, it greys out what cannot be connected. When the users starts dragging the arrow for the output of a node, the compatible inputs get highlighted and the incompatible outputs get greyed out.
(e.g., if a float is plugged into a texture input, do we error out or try to convert it?)
Q9: Do settings inside a node (like Strength or Texture) also need to declare their own input/output types — or should they inherit from the node? They need to declare their own types. Each property of a node can accept its own input, they don't all share the input of their parent node. You can't have multiple properties of a single node have more then one input value - for example the opacity adjustment in a blur node can’t take input from multiple sources. It needs to get its input from a single output value from another node. 

🎨 4. UX + Visual Model Behavior
Q10: Should each node parameter (like “Strength”) be visualized as:
A slider that doubles as a link port? No, itll always be separate. I’d imagine the way that the adjustment ui looks is eg; (inside a blur filter node)  Strength to the right of the labeled adjustment is a slider, one that is similar to Blenders sliders for example the sprite threshold setting in depth of field settings. Then on the further left of strength is a circle that indicates where the connection should be made. 


A mini UI block with input port on the left, label, and value? Correct, similar to how i describe it above. 


Should it show both its default value AND a connected node’s override? No, just either one or the other. 


Q11: When a parameter is linked to another node, what happens to the manual slider control?
 (Should it be greyed out, replaced, or blended?) No, it should be disable, you should have to unlink a properties/adjustment input first, before manual editing it.
