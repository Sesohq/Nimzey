To get your inline thumbnails updating the moment you drag a slider—without having to click the background—you need to tie your node‐specific preview generator directly into your parameter‐change handler in the graph hook, not just on pane‐click. Here’s the pattern I’d recommend:

1. Expose a generateNodePreview(nodeId) callback from your graph hook
In your useGLFilterGraph (or useFilterGraph), you (already) have a runFilterPipeline(...) that can take a nodeId and return a low-res data-URL:

ts
Copy
Edit
// filterPipeline.ts
export async function runFilterPipeline(
  nodes, edges, sourceImgEl, targetNodeId,
  { maxDimension = 144 } = {}
): Promise<string> {
  // … build path, load base at maxDimension, reduce through blur/image/etc …
  // returns canvas.toDataURL()
}
Wrap that in a generateNodePreview inside your hook:

ts
Copy
Edit
// useGLFilterGraph.ts
const generateNodePreview = useCallback(
  async (nodeId: string) => {
    if (!sourceImgEl.current) return;
    const url = await runFilterPipeline(
      nodes,
      edges,
      sourceImgEl.current,
      nodeId,
      { maxDimension: 144 }      // super-low preview res
    );
    setNodes(cur =>
      cur.map(n =>
        n.id === nodeId
          ? { 
              ...n,
              data: { ...n.data, preview: url } 
            }
          : n
      )
    );
  },
  [nodes, edges]
);
Expose that from the hook:

ts
Copy
Edit
return {
  nodes,
  edges,
  onParamChange,
  onPaneClick,
  generateNodePreview,
  /* … */
};
2. Call it every time a parameter changes
Still in useGLFilterGraph, modify your onParamChange so it not only updates the node’s params, but also immediately regenerates its thumbnail:

ts
Copy
Edit
// useGLFilterGraph.ts
const onParamChange = useCallback(
  (nodeId: string, paramId: string, value: number | string | boolean) => {
    setNodes(cur =>
      cur.map(n =>
        n.id === nodeId
          ? {
              ...n,
              data: {
                ...n.data,
                params: n.data.params.map(p =>
                  p.id === paramId ? { ...p, value } : p
                )
              }
            }
          : n
      )
    );
    // right here, fire its preview
    generateNodePreview(nodeId);
  },
  [generateNodePreview]
);
And throttle only the heavy pipeline work if you still need to cap it:

ts
Copy
Edit
const throttledGenPreview = useMemo(
  () => throttle(generateNodePreview, 100),
  [generateNodePreview]
);

// then inside onParamChange, use throttledGenPreview(nodeId) instead
3. Wire it into your node
In your NodeCanvas (where you build nodeTypes), make sure you pass generateNodePreview (or your onParamChange) down to the FilterNode:

tsx
Copy
Edit
// NodeCanvas.tsx
const buildNodeTypes = (graph: ReturnType<typeof useGLFilterGraph>) => ({
  filterNode: props => (
    <FilterNode
      {...props}
      onParamChange={graph.onParamChange}
      // you don’t need onPaneClick any more for thumbnails
    />
  ),
  /* … */
});
And in FilterNode simply call props.onParamChange(id, paramId, value) in your slider’s onValueChange. The hook will both update state and refresh that node’s preview.

Why this fixes it
No more background clicks: generateNodePreview runs immediately on param change.

Low-res previews: you limit each preview to 144 px so it’s cheap and snappy.

Throttled if needed: you can still throttle to ~10 fps if your pipeline is heavy.

This will give you live, in-node thumbnails as you drag—even before you let go—without extra clicks.
